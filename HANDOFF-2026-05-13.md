## Open Issues (Two UX Problems — User's Last Message)

The user reported two problems with the current Debug build (Merlin is still running, session active):

### Problem 1: Content order

**User said**: "The updates appear at the top of the screen, and the executions spawn below."

Currently: text div renders ABOVE tool group. During a long agentic run, the agent's text is at the top of the bubble, tool calls accumulate below. The user has to auto-scroll to the bottom to see the latest tool execution; the text context is off-screen above.

**Likely fix**: Revert order back to `\(toolGroup)\(textDiv)` (tools first, text below). The duplicate-element bug that made this confusing before is now fixed — the `appendChunk` fallback that was creating second bubbles is gone. With tools before text:
- Tool call rows (compact, collapsed by default) show at top as the agent works
- Text summary builds below them
- Auto-scroll bottom shows end of text

Also revert `.tool-group { margin-top: 8px }` → `margin-bottom: 8px`.

### Problem 2: Window resize

**User said**: "Resizing the outer window does not behave as expected."

Not yet diagnosed. The WKWebView is in `.frame(maxWidth: .infinity, maxHeight: .infinity)` in SwiftUI. Known candidates:
- WKWebView internal scroll view doesn't reflow when outer frame changes
- Missing `html, body { width: 100%; height: 100%; }` in CSS (currently only sets background/font)
- The `#messages` div uses `min-height: 100vh` which is a viewport-relative unit — may not update on resize

**Investigation starting point**: `ConversationWebView.swift:53` (`updateNSView`) — there is no frame/size tracking. Consider adding:
1. `html, body { width: 100%; }` to the CSS (line ~259 in renderer)
2. In `updateNSView`, when a resize is detected, call `webView.evaluateJavaScript("window.dispatchEvent(new Event('resize'))")` to force web relayout
3. Or subclass WKWebView to override `layout()` — most robust but more code
